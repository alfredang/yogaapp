# Install an iOS app on a physical iPhone/iPad

Build a native app from an Xcode project and install it onto a real device, **wired or
wireless**, entirely from the command line. Targets Xcode 15+ / iOS 17+ where
`xcrun devicectl` is the supported tool (it replaced the deprecated `ios-deploy` and
`instruments`).

This is **development install** (free Apple Development signing), not TestFlight/App Store.
For App Store / TestFlight distribution use the `app-store-submission` skill instead.

## Prerequisites (one-time)

- macOS with Xcode + command line tools (`xcode-select -p` should resolve).
- The device is **paired** and **trusted** with this Mac (tap **Trust** on the device the
  first time you connect it via USB and enter the passcode).
- **Developer Mode** is ON on the device: **Settings → Privacy & Security → Developer Mode
  → On**, then reboot. (Required on iOS 16+ to run development builds.)
- A signing team is available. For an XcodeGen project this is `DEVELOPMENT_TEAM` in
  `project.yml`; otherwise it's set in the target's Signing & Capabilities. `xcodebuild
  -allowProvisioningUpdates` will create/refresh the development provisioning profile
  automatically.

## Workflow

### Step 1 — Find the device and its UDID

```bash
xcrun devicectl list devices
```

Look for your device in the table and copy its **identifier/UDID**. State should be
`available (paired)` (or `connected`). Note the **Identifier** column.

> `devicectl` shows a CoreDevice UUID. For `xcodebuild -destination 'id=...'` you want the
> **hardware UDID** (the 25-char `00008XXX-XXXXXXXXXXXXXXXX` form). Get it with:
> ```bash
> xcrun xctrace list devices 2>&1 | grep -v Simulator
> ```
> Both UDID forms work with `devicectl device install`, but `xcodebuild -destination` is
> most reliable with the hardware UDID from `xctrace`. If a device shows **Offline** in
> `xctrace` but `available (paired)` in `devicectl`, prefer `devicectl` — it can wake the
> tunnel.

If no project path is known yet, find it:

```bash
find . -maxdepth 3 \( -name "*.xcworkspace" -o -name "*.xcodeproj" \) -not -path '*/build/*'
```

Use `-workspace X.xcworkspace` when a workspace exists (CocoaPods/SPM), otherwise
`-project X.xcodeproj`. The scheme usually matches the app target name; list schemes with
`xcodebuild -list -project X.xcodeproj`.

### Step 2 — Build + sign for the device

```bash
xcodebuild \
  -project AIExams.xcodeproj \        # or -workspace App.xcworkspace
  -scheme AIExams \
  -configuration Debug \
  -destination 'id=<HARDWARE_UDID>' \
  -allowProvisioningUpdates \
  -derivedDataPath build
```

- `-allowProvisioningUpdates` lets Xcode register the device and mint the dev profile.
- `-derivedDataPath build` puts the product at a predictable path (next step).
- End on `** BUILD SUCCEEDED **`. The signed app lands at:
  `build/Build/Products/Debug-iphoneos/<AppName>.app`

If the project is generated by XcodeGen and the `.xcodeproj` looks stale, regenerate first:
`xcodegen generate`.

### Step 3 — Install onto the device

```bash
xcrun devicectl device install app \
  --device <UDID> \
  build/Build/Products/Debug-iphoneos/<AppName>.app
```

Success prints `App installed:` with the `bundleID` and an `installationURL`. The icon
appears on the home screen.

Optionally launch it immediately:

```bash
xcrun devicectl device process launch --device <UDID> <bundle.id>
```

### Step 4 — First-launch trust (free / personal dev signing)

If the app won't open and shows "Untrusted Developer", trust the cert on the device:
**Settings → General → VPN & Device Management → (your Apple Development cert) → Trust.**
Then tap the icon again. (Apps signed with a paid Apple Developer Program cert + a
registered device usually skip this.)

## Wired vs wireless

The commands above are **identical** for both — the only difference is how the device is
connected/reachable.

**Wired (USB):** plug in, unlock, tap **Trust** if prompted. `devicectl list devices`
shows it as `connected`/`available`. Most reliable; use this for the first pairing and for
large/first installs.

**Wireless (Wi-Fi):** enable it once while plugged in:

1. Connect the device via USB and unlock it.
2. In Xcode: **Window → Devices and Simulators → (select device) → check "Connect via
   network".** (One-time per Mac+device.) The Mac and device must be on the **same
   network**.
3. Unplug. The device now appears in `xcrun devicectl list devices` over the network, and
   the exact same Step 2/Step 3 commands work without a cable.

There is no separate "wireless install" flag — once "Connect via network" is enabled,
`devicectl` reaches the device over the LAN automatically. Wireless installs are slower and
the device must stay awake/unlocked and on the same Wi-Fi.

## Quick reference (one device, hardware UDID known)

```bash
UDID=00008140-XXXXXXXXXXXXXXXX
xcodebuild -project App.xcodeproj -scheme App -configuration Debug \
  -destination "id=$UDID" -allowProvisioningUpdates -derivedDataPath build
xcrun devicectl device install app --device "$UDID" \
  build/Build/Products/Debug-iphoneos/App.app
```

## Troubleshooting

| Symptom | Fix |
|---|---|
| Device `Offline` in `xctrace` but needed for `xcodebuild` | Use the `devicectl` UDID for the `-destination 'id=...'`, or unlock the device and re-run. `devicectl list devices` showing `available (paired)` is enough to install. |
| `Unable to install … This app cannot be installed because its integrity could not be verified` | Untrusted cert — do Step 4 (trust the developer profile). |
| `The developer disk image could not be mounted` / DDI errors | Update Xcode to match the device's iOS version; reconnect; reboot the device. `devicectl` auto-mounts the DDI on a fresh connection. |
| `No profiles for 'com.x.y' were found` | Add `-allowProvisioningUpdates`; ensure `DEVELOPMENT_TEAM` is set and you're signed into the Apple ID in Xcode (Settings → Accounts). |
| `Developer Mode disabled` | Enable Settings → Privacy & Security → Developer Mode, reboot. |
| Device not listed at all | Re-plug USB, unlock, tap **Trust**; for Wi-Fi confirm "Connect via network" is checked and both devices share a network. |
| Free Apple ID: app stops working after ~7 days | Personal-team dev certs expire in 7 days — rebuild + reinstall, or use a paid Developer Program account. |

## Notes

- `devicectl` is the modern replacement for `ios-deploy`/`instruments -w`. If on Xcode 14
  or earlier, fall back to `xcrun simctl` (simulator only) or third-party `ios-deploy`.
- Installing a Debug build over an existing App Store / TestFlight build of the same bundle
  id replaces it; user data in the app container is typically wiped on signing-identity
  change.
- To uninstall: `xcrun devicectl device uninstall app --device <UDID> <bundle.id>`.
